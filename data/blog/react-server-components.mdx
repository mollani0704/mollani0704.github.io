# 리액트 서버 컴포넌트

React의 렌더링 방식은 단순히 "클라이언트에서 그린다"에서 끝나지 않는다. 초기 CSR에서 시작해 SSR, Hybrid Rendering을 거쳐 지금은 React Server Components(RSC)까지 진화했다. 이 글에서는 그 흐름과 RSC가 왜 필요한지 정리해본다.

## 1. 초기 React, CSR(Client Side Rendering)

초기 React 앱은 기본적으로 **CSR 방식**이었다.

동작 구조는 다음과 같다.

```tsx
서버 -> 빈 HTML 전달
브라우저 -> JS 실행
React -> DOM 생성
```

예를 들면 서버는 이런 식의 HTML만 내려준다.

```html
<body>
  <div id="root"></div>
</body>
```

실제 화면은 브라우저가 JavaScript를 실행한 뒤에 React가 만들어낸다.

### CSR의 장점

- 인터랙티브 UI를 구현하기 좋다.
- SPA 구조에 잘 맞는다.
- 페이지 새로고침 없이 화면 전환이 가능하다.
- 서버가 UI를 완성해서 보내지 않아도 되므로 서버 부담이 줄어든다.

### CSR의 단점

대표적인 문제는 세 가지다.

1. 초기 로딩이 느릴 수 있다.
2. API 요청이 워터폴 구조로 이어지기 쉽다.
3. SEO에 불리하다.

특히 컴포넌트가 마운트된 뒤 데이터를 가져오는 구조에서는 아래처럼 요청이 연쇄적으로 이어진다.

```tsx
컴포넌트 -> 데이터 요청
```

그 결과 실제 네트워크 흐름은 이런 식이 된다.

```tsx
API 요청
↓
다음 요청
↓
다음 요청
```

이런 **network waterfall**은 첫 화면이 보이기까지의 시간을 늘린다.

## 2. 해결 시도, SSR(Server Side Rendering)

CSR의 문제를 줄이기 위해 등장한 방식이 **SSR**이다.

동작 구조는 다음과 같다.

```tsx
브라우저 -> 서버 요청
서버 -> React 렌더링
서버 -> 완성된 HTML 전달
```

### SSR의 장점

```tsx
초기 로딩 빠름
SEO 좋음
```

### SSR의 단점

```tsx
서버 부담 증가
동적 사이트 확장 어려움
```

결국 CSR과 SSR은 서로 장단점이 뚜렷한 **trade-off 관계**였다.

## 3. 현재, Hybrid 방식

그래서 나온 현실적인 해법이 **CSR + SSR 혼합 방식**이다.

대표적인 프레임워크는 다음과 같다.

- Next.js
- Remix

일반적인 전략은 아래와 같다.

```tsx
초기 페이지 -> SSR
이후 인터랙션 -> CSR
```

이 접근은 실용적이지만, 여전히 몇 가지 문제가 남아 있다.

- JavaScript 번들 크기가 커지기 쉽다.
- 데이터 요청 구조가 여전히 복잡하다.
- 서버와 클라이언트의 책임 경계가 모호해진다.

## 4. 미래, React Server Components

이런 문제를 해결하기 위해 등장한 것이 **React Server Components**다.

핵심 아이디어는 간단하다.

```tsx
컴포넌트 일부는 서버에서 실행
컴포넌트 일부는 브라우저에서 실행
```

즉, 컴포넌트를 다음 두 종류로 나눈다.

```tsx
Server Component
Client Component
```

## 5. Server Component vs Client Component

### Server Component

Server Component는 서버에서 실행된다.

가능한 작업은 다음과 같다.

```tsx
DB 접근
API 호출
무거운 계산
```

그리고 중요한 특징이 하나 있다.

```tsx
JS 번들로 클라이언트에 보내지 않음
```

즉, 브라우저가 굳이 실행할 필요가 없는 로직은 클라이언트 번들에 포함되지 않는다.

### Client Component

Client Component는 브라우저에서 실행된다.

이쪽은 사용자 인터랙션을 담당한다.

```tsx
useState
useEffect
event handler
click
```

즉, 상태 관리나 이벤트 처리처럼 브라우저에서 반드시 실행되어야 하는 기능은 Client Component에 둔다.

## 6. RSC의 핵심 특징

### 1. JavaScript 번들 감소

Server Component는 브라우저로 보내지지 않는다.

```tsx
JS bundle ↓
```

이는 초기 로딩 성능과 전송량 측면에서 큰 이점이 된다.

### 2. 데이터 fetch를 서버에서 처리

예를 들어 다음 코드는 브라우저가 아니라 **서버에서 실행**된다.

```tsx
export default async function Page() {
  const users = await db.user.findMany()
}
```

즉, 데이터를 가져오기 위해 클라이언트가 별도 요청을 시작할 필요가 없다.

### 3. API 레이어가 꼭 필요하지 않음

기존 구조는 보통 이랬다.

```tsx
React
↓
API
↓
DB
```

하지만 RSC에서는 다음처럼 단순화할 수 있다.

```tsx
React
↓
DB
```

물론 모든 경우에 API가 사라지는 것은 아니지만, 서버 컴포넌트가 직접 서버 리소스에 접근할 수 있다는 점이 핵심이다.

### 4. 스트리밍 가능

RSC에서는 서버가 단순한 HTML만 보내는 것이 아니라 컴포넌트 단위의 데이터를 스트리밍할 수 있다.

```tsx
HTML
```

이 아니라

```tsx
Component data stream
```

을 보내는 방식에 가깝다.

그래서 UI를 한 번에 다 보여주는 대신, **준비된 부분부터 점진적으로 렌더링**할 수 있다.

## 7. RSC의 동작 구조

RSC는 전통적인 의미의 완성된 HTML만 내려주는 방식이 아니다. 서버는 **컴포넌트 데이터 스트림**을 보내고, 브라우저는 이를 받아 Client Component와 결합해 화면을 완성한다.

개념적으로 보면 이런 식이다.

```json
{
  "type": "div",
  "children": [
    { "type": "text", "value": "Hello" },
    { "type": "ClientComponent", "props": {} }
  ]
}
```

브라우저는 이 데이터를 기반으로 필요한 클라이언트 컴포넌트를 붙여 최종 UI를 구성한다.

## 8. RSC의 핵심 목표

RSC가 해결하려는 문제는 명확하다.

```tsx
CSR 문제
↓
JS 번들
워터폴 요청
초기 로딩
```

결국 목표는 역할 분리다.

```tsx
서버 -> 데이터 + 렌더링
브라우저 -> 인터랙션
```

데이터 처리와 무거운 렌더링은 서버가 맡고, 사용자 상호작용은 브라우저가 맡게 만드는 것이다.

## 9. 페이지 로딩 타임라인의 변화

기존 CSR의 로딩 흐름은 대체로 다음과 같다.

```tsx
HTML
↓
JS 다운로드
↓
React 실행
↓
데이터 요청
↓
렌더링
```

반면 RSC는 아래와 같은 흐름을 지향한다.

```tsx
HTML + 데이터
↓
즉시 UI
↓
필요한 JS만 다운로드
```

이 구조 덕분에 사용자는 더 빠르게 콘텐츠를 볼 수 있고, **Time to Content**도 개선된다.

## 마무리

React Server Components는 단순히 "서버에서 렌더링한다"는 개념을 넘어서, 서버와 클라이언트가 각자 잘하는 일을 나눠 맡게 하려는 시도다. CSR의 번들 부담, 데이터 요청 워터폴, 초기 로딩 지연 문제를 줄이기 위한 React의 다음 단계라고 볼 수 있다.

Next.js App Router를 쓰고 있다면 이미 RSC 기반 위에서 개발하고 있는 셈이다. 그래서 `use client`가 왜 필요한지, 어떤 로직을 서버 컴포넌트로 남겨야 하는지를 이해하는 것이 점점 더 중요해지고 있다.
