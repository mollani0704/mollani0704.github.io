## JavaScript 날짜 문제와 Temporal API 정리

해당 블로그 글의 핵심 내용은 **JavaScript의 기존 `Date` 객체가 시간대를 포함한 실제 세계의 날짜/시간 문제를 제대로 표현하기 어렵다는 점**이며, 이를 해결하기 위해 **Temporal API가 제안되었다는 것**이다.

사용자가 이해한 것처럼 `new Date()`는 시간대를 포함한 다양한 상황을 안전하게 처리하기 어렵다. 특히 서로 다른 국가나 시간대에서 데이터를 저장하고 다시 표시하는 과정에서 예상하지 못한 문제가 발생할 수 있다.

이를 이해하기 위해서는 먼저 **시간을 표현하는 방식**을 구분해야 한다.

## 1. 시간 표현의 세 가지 개념

컴퓨터에서 시간을 다룰 때는 크게 세 가지 개념이 존재한다.

### 1) 절대적인 순간 (Instant)

전 세계 어디에서 보더라도 동일한 하나의 시점을 의미한다.

예시

- 어떤 결제가 실제로 발생한 순간
- 서버 로그가 기록된 순간

컴퓨터 시스템에서는 이 값을 보통 **UTC 기준 timestamp**로 저장한다.

예시

```txt
2026-03-17T00:00:00Z
```

여기서 `Z`는 UTC 시간을 의미한다.

### 2) 현지 시간 (Local Time)

사람이 실제로 사용하는 시계 기준의 시간이다.

예시

- 서울 오전 10시
- 마드리드 오후 3시
- 뉴욕 오전 9시

같은 순간이라도 지역에 따라 다른 시간으로 표현된다.

### 3) 시간대 (Time Zone)

현지 시간이 **어떤 지역의 규칙을 따르는지**를 의미한다.

대표적인 예

```txt
Asia/Seoul
Europe/Madrid
America/Los_Angeles
```

시간대는 단순한 시간 차이만 의미하지 않는다.

여기에는 **서머타임(DST)과 같은 규칙**도 포함된다.

따라서 시간대를 포함해야만 실제 현지 시간의 의미가 정확하게 표현된다.

## 2. UTC, CET, ISO 시간 형식

### UTC (협정 세계시)

UTC는 전 세계가 공통적으로 사용하는 기준 시간이다.

각 지역은 UTC 기준으로 몇 시간 차이가 나는지로 표현된다.

예시

| 지역     | 시간             |
| -------- | ---------------- |
| 서울     | UTC+9            |
| 마드리드 | UTC+1 또는 UTC+2 |
| LA       | UTC-8 또는 UTC-7 |

마드리드처럼 일부 지역은 **서머타임 때문에 계절에 따라 시간이 바뀐다.**

### CET

CET는 **Central European Time**이다.

유럽 중부 표준시를 의미한다.

일반적으로

```txt
CET = UTC +1
```

그러나 여름에는 **CEST (UTC+2)** 로 변경된다.

### ISO 날짜 형식

ISO는 날짜를 표현하는 표준 문자열 형식이다.

예시

```txt
2024-07-20T08:30:00.000Z
```

또는

```txt
2024-07-20T10:30:00+02:00
```

여기서

- `Z`는 UTC 시간
- `+02:00`은 UTC보다 2시간 빠른 시간

을 의미한다.

그러나 이 정보만으로는 **실제 시간대(Time Zone)** 를 완전히 알 수 없다.

예를 들어 `+02:00`이라는 오프셋은 여러 지역에서 동일하게 사용될 수 있기 때문이다.

## 3. JavaScript `Date` 객체의 문제

JavaScript의 `Date` 객체는 내부적으로 **하나의 timestamp(밀리초)** 값으로 시간을 저장한다.

즉, 다음과 같은 방식이다.

```txt
1970-01-01 UTC 이후 몇 밀리초가 지났는가
```

따라서 `Date`는 실제로는 **하나의 순간만 저장**한다.

예시

```js
const date = new Date('2024-07-20T10:30:00')
```

이 코드는 다음과 같은 문제를 가진다.

1. 어떤 시간대를 기준으로 작성된 시간인지 명확하지 않다.
2. 실행되는 환경의 로컬 시간대에 따라 결과가 달라질 수 있다.

예를 들어 마드리드 기준 10:30이라는 시간을 저장했다고 가정하면

| 지역     | 표시 시간 |
| -------- | --------- |
| 마드리드 | 10:30     |
| UTC      | 08:30     |
| 서울     | 17:30     |

같은 순간이지만 지역에 따라 다른 시간으로 표시된다.

## 4. 마드리드 → 시드니 여행 예시

블로그에서 설명하는 대표적인 문제 상황은 다음과 같다.

1. 사용자는 마드리드에 거주한다.
2. 시드니 여행을 간다.
3. 시드니 현지 시간 아침 8시에 커피를 결제한다.
4. 은행 시스템은 결제 순간을 UTC timestamp로 저장한다.
5. 사용자가 마드리드로 돌아와 거래내역을 확인한다.

이 경우 결제 시간은 다음과 같이 보일 수 있다.

- 시드니: 오전 8시
- 마드리드: 전날 밤 또는 새벽

사용자는 다음과 같은 혼란을 겪을 수 있다.

- "이 시간에 결제를 했던가?"
- "나는 그때 자고 있었는데?"
- "부정 결제가 아닌가?"

이 문제는 **결제가 발생한 현지 시간 맥락이 사라졌기 때문**이다.

## 5. 오프셋만으로는 해결되지 않는 이유

많은 경우 다음과 같이 생각한다.

```txt
UTC +9
UTC -8
```

이러한 오프셋만 저장하면 충분하지 않은가?

그러나 실제 시간대는 단순한 숫자가 아니다.

예시

```txt
Europe/Madrid
America/Los_Angeles
```

이러한 시간대에는 다음 정보가 포함된다.

- 서머타임
- 역사적 시간 변경
- 지역별 시간 규칙

따라서 **정확한 시간 처리를 위해서는 오프셋보다 시간대 ID가 중요하다.**

## 6. Temporal API의 등장

Temporal API는 기존 `Date` 객체의 문제를 해결하기 위해 설계된 새로운 API이다.

핵심 철학은 다음과 같다.

**날짜, 시간, 시간대, 순간을 하나의 객체로 처리하지 않고 각각 분리한다.**

대표적인 객체는 다음과 같다.

| 객체                     | 의미                 |
| ------------------------ | -------------------- |
| `Temporal.Instant`       | 절대적인 순간        |
| `Temporal.PlainDate`     | 날짜만               |
| `Temporal.PlainTime`     | 시간만               |
| `Temporal.PlainDateTime` | 날짜 + 시간          |
| `Temporal.ZonedDateTime` | 날짜 + 시간 + 시간대 |
| `Temporal.Duration`      | 시간 길이            |

## 7. 가장 중요한 객체: ZonedDateTime

`Temporal.ZonedDateTime`은 다음 정보를 모두 포함한다.

- 날짜
- 시간
- 시간대

예시

```jsx
const zdt = Temporal.ZonedDateTime.from('1996-12-19T16:39:57-08:00[America/Los_Angeles]')
```

이 객체는 다음 정보를 모두 가진다.

- 날짜
- 시간
- 오프셋
- 실제 시간대 규칙

따라서 DST와 같은 문제를 정확하게 처리할 수 있다.

## 8. Date vs Temporal 비교

### 기존 Date

```js
const d = new Date()
```

문제점

- 어떤 시간 기준인지 불명확
- 시간대 개념이 약함
- 날짜와 시간 의미가 혼합됨

### Temporal

```jsx
const date = Temporal.PlainDate.from('2026-03-17')

const instant = Temporal.Instant.from('2026-03-17T00:00:00Z')

const zdt = Temporal.ZonedDateTime.from('2026-03-17T09:00:00+09:00[Asia/Seoul]')
```

장점

- 데이터의 의미가 명확하다.
- 시간대 처리가 안전하다.
- DST와 같은 복잡한 상황을 정확히 처리할 수 있다.

## 9. 언제 어떤 타입을 사용하는가

### 날짜만 중요한 경우

```txt
Temporal.PlainDate
```

예

- 생일
- 기념일
- 청구 마감일

### 정확한 순간이 중요한 경우

```txt
Temporal.Instant
```

예

- 결제 발생 시점
- 서버 로그
- 이벤트 기록

### 현지 시간대가 중요한 경우

```txt
Temporal.ZonedDateTime
```

예

- 회의 일정
- 비행기 출발 시간
- 호텔 체크인

## 10. 핵심 요약

이 글의 핵심 메시지는 다음과 같다.

**JavaScript의 `Date` 객체는 하나의 절대적인 순간을 표현하는 데에는 적합하지만, 실제 세계의 날짜와 시간대 문제를 표현하기에는 한계가 있다.**

Temporal API는

- 날짜
- 시간
- 시간대
- 순간

을 명확하게 구분하여 이러한 문제를 해결하려는 새로운 JavaScript 표준이다.
